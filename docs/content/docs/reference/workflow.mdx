## Front matter schema

先记住一个总规则：

- `tracker.*`：任务来源和状态同步
- `provider.*`：执行后端
- `codex.*`：`provider.kind: codex` 时的 provider 细节

如果你是为了接 GitHub Projects v2，看的是 `tracker.kind: github` 这一段，不是 `provider.kind`。当前稳定 provider 只有 `codex`。

### `tracker`

```yaml
tracker:
  kind: linear
  endpoint: https://api.linear.app/graphql
  api_key: $LINEAR_API_KEY
  project_slug: your-project-slug
  active_states: ["Todo", "In Progress"]
  terminal_states: ["Closed", "Cancelled", "Canceled", "Duplicate", "Done"]
  page_size: 50
  timeout_ms: 30000
```

说明：

Linear：

- `tracker.kind: linear`
- 使用字段：`endpoint`、`api_key`、`project_slug`
- 省略 `tracker.kind` 时默认仍然是 `linear`
- tracker 字段支持 `$ENV_VAR` 展开

GitHub Projects v2：

- `tracker.kind: github`
- 使用字段：`endpoint`、`token`、`project_owner`、`project_number`、`repository`
- 可选字段：`state_field`
- 默认值：`endpoint: https://api.github.com/graphql`、`state_field: Status`
- 当前限制：一个 workflow 只支持一个 repository，且 `tracker.state_field` 必须已存在、必须是 single-select，option 名称必须与工作流状态名完全一致

GitHub 这里的读取范围也要注意：

- `project_owner + project_number` 指向一个具体的 Project v2
- Synclax 先读取这个 Project 的 items
- 然后只保留 `tracker.repository` 对应仓库的 `Issue`
- `DraftIssue`、`PullRequest` 和其他仓库的 item 不会进入调度

GitHub Projects v2 示例：

```yaml
tracker:
  kind: github
  endpoint: https://api.github.com/graphql
  token: $GITHUB_TOKEN
  project_owner: your-org-or-user
  project_number: 1
  repository: your-org-or-user/your-repo
  state_field: Status
  active_states: ["Todo", "In Progress"]
  terminal_states: ["Done", "Closed", "Cancelled", "Canceled", "Duplicate"]
```

### `provider`

```yaml
provider:
  kind: codex
```

当前稳定支持的 `provider.kind` 只有：

- `codex`

省略 `provider.kind` 时会默认使用 `codex`。如果你想避免和 GitHub tracker 概念混淆，建议在 workflow 里显式写出来。

### `polling`

```yaml
polling:
  interval_ms: 30000
```

### `workspace`

```yaml
workspace:
  root: ~/.cache/symphony_workspaces
```

### `hooks`

```yaml
hooks:
  after_create: |
    git clone git@github.com:your-org/your-repo.git .
  before_run: |
    make gen
  after_run: |
    echo "done"
  before_remove: |
    echo "cleanup"
  timeout_ms: 60000
```

### `agent`

```yaml
agent:
  max_concurrent_agents: 3
  max_turns: 10
  max_retry_backoff_ms: 300000
```

### `codex`

```yaml
codex:
  command: codex app-server
  read_timeout_ms: 5000
  turn_timeout_ms: 3600000
  stall_timeout_ms: 300000
```

## Dynamic tools

当前支持的动态工具：

- `linear_graphql`
  使用 Symphony 的 Linear auth 执行原始 GraphQL 查询 / mutation。
  要求：`tracker.kind: linear`，`tracker.endpoint` 指向 Linear GraphQL（默认 `https://api.linear.app/graphql`），并且 `tracker.api_key` 已设置或环境里存在 `LINEAR_API_KEY`。

- `github_graphql`
  使用 Symphony 的 GitHub auth 执行原始 GraphQL 查询 / mutation。
  要求：`tracker.kind: github`，`tracker.endpoint` 指向 GitHub GraphQL（默认 `https://api.github.com/graphql`），并且 `tracker.token` 已设置或环境里存在 `GITHUB_TOKEN`。

## Prompt template（Liquid）

模板可用变量：

- `attempt`
- `issue.id`
- `issue.identifier`
- `issue.title`
- `issue.description`
- `issue.state`
- `issue.url`
- `issue.labels`
- `issue.blocked_by`

示例：

```md
You are working on an issue.

- Title: {{ issue.title }}
- Identifier: {{ issue.identifier }}
- URL: {{ issue.url }}
- Attempt: {{ attempt }}
```

<Callout type="warning">
  启用了 strict variables。引用不存在的变量会直接导致模板渲染失败。
</Callout>
